# 09. 代码抽象能力与系统设计

## 为什么这条线同样非常值钱

很多人做完作业，会觉得自己最大的收获是：

- 会写卷积
- 会写 RNN
- 会写 attention

这些当然重要，但如果从更长期的工程能力看，还有另一层更隐蔽也更值钱的训练：

> 如何把复杂模型组织成清晰、可复用、可调试、可扩展的代码系统。

这就是代码抽象能力与系统设计能力。

CS231n 的三份作业，在这方面其实是一条非常完整的进阶路径。

## 一、assignment1：从零建立“层”的抽象

assignment1 的重要价值，不只是让你手写 forward / backward，而是让你第一次系统地见到：

- 每层有清晰输入输出
- 每层可以缓存中间量
- backward 与 forward 一一对应
- 参数用统一字典管理

这是非常经典的软件抽象训练。

### 1. 层的接口

例如一个 affine 层会自然形成这样的接口：

```python
out, cache = affine_forward(x, w, b)
dx, dw, db = affine_backward(dout, cache)
```

这带来了两个好处：

- 调试边界清晰
- 可以自由组合层

### 2. 参数与计算分离

你会把：

- `W`, `b` 作为参数
- `x`, `out`, `cache` 作为计算过程对象

这一步其实就是工程里的“状态分类”。

## 二、assignment2：从单层抽象升级到模块抽象

assignment2 开始出现更高层封装：

- `affine_relu_forward`
- `conv_relu_pool_forward`
- 各种网络类

这意味着你开始学会：

- 把多个常见操作组合成一个可复用 block
- 在 block 上继续拼装大网络

这是一个很大的进步，因为真正的大模型几乎不可能只靠原子操作直接手拼。

## 三、assignment3：从“手写层”走向现代系统化工程

assignment3 引入 PyTorch 后，代码组织开始更接近真实项目：

- `nn.Module`
- `Dataset`
- `DataLoader`
- `Trainer` / `Solver`
- utility functions
- task-specific pipelines

这其实是在训练你一种更现代的系统视角：

- 数据系统
- 模型系统
- 训练系统
- 评估系统

这些子系统相互协作，但职责分离。

## 四、你应该形成的 5 个核心抽象观念

## 1. 接口优先

一个模块最重要的不是“内部多聪明”，而是：

- 输入是什么
- 输出是什么
- 它承诺做什么

如果接口稳定，内部实现以后可以替换。

例如：

- `forward(x)` 返回什么
- `loss(x, y)` 返回哪些对象
- `__getitem__` 返回怎样的样本结构

## 2. 单一职责

一个模块最好只做一件主要事情。

例如：

- `Dataset` 负责取数据
- `Model` 负责 forward
- `Optimizer` 负责更新
- `Trainer` 负责训练循环

如果这些职责混在一起，代码会越来越难维护。

## 3. 显式的数据流

好的深度学习代码应该让人容易看清：

- 数据从哪里来
- 经过哪些变换
- 在哪里变成 loss
- 在哪里更新参数

也就是说，数据流要清楚。

## 4. 参数、状态、超参数分开

这是非常重要的系统设计意识。

### 参数

模型要学习的量，例如：

- `weight`
- `bias`
- `gamma`
- `beta`

### 状态

训练过程中维护，但不一定直接梯度更新的量，例如：

- BatchNorm running mean / var
- optimizer momentum
- random seed state

### 超参数

你手动配置的控制项，例如：

- learning rate
- dropout rate
- num_heads
- hidden_dim

这三类概念混了，代码就会很乱。

## 5. 复用优先于复制

如果你发现自己在多处重复相似逻辑，就应该考虑：

- 能不能提成函数？
- 能不能提成 block？
- 能不能提成公共工具模块？

这不是为了“写得高级”，而是为了降低未来改错和扩展的成本。

## 五、三份作业里最值得吸收的结构经验

## 1. `model.params` 的思想

在 NumPy 作业里，把参数集中到一个字典里，是很经典也很实用的设计。

优点：

- 更新统一
- 命名清楚
- 便于调试和保存

这和 PyTorch 里 `nn.Module` 自动注册参数，在思想上是一致的。

## 2. `cache` 的思想

`cache` 本质上是：

- 为 backward 保留 forward 的必要上下文

这件事训练你建立“前后依赖”的意识。

即便以后用 autograd，这种思维仍然有用，因为很多工程设计都需要明确哪些中间结果要保留。

## 3. Solver / Trainer 的思想

把训练循环从模型中抽出来，是非常重要的架构选择。

好处：

- 模型更纯粹
- 训练逻辑可复用
- 更容易比较不同模型

这是从“能写一个模型”走向“能管理一组实验”的关键一步。

## 4. Dataset / DataLoader 的思想

assignment3 里的数据系统让你看到：

- 数据读取
- 增强
- batch 组织
- shuffle
- 多样本结构

都应该有明确位置，而不是散落在训练循环里。

## 六、你以后写任何项目时都可以用的系统骨架

一个很通用的深度学习项目结构可以是：

```text
project/
  data/
    dataset.py
    transforms.py
  models/
    blocks.py
    backbone.py
    heads.py
    model.py
  losses/
    loss.py
  train/
    trainer.py
    eval.py
  utils/
    metrics.py
    logging.py
    checkpoint.py
  configs/
    default.yaml
  main.py
```

这并不是唯一结构，但体现了一个核心原则：

> 按职责分层，而不是把所有逻辑堆进一个文件。

## 七、如何避免“作业代码能跑，但项目代码很乱”

很多人从作业过渡到项目时，会遇到这个问题。

建议你注意下面几点：

### 1. 不要把实验代码、可视化代码、模型代码全混在 notebook 主体里

能封装的尽量封装。

### 2. 不要把 shape 约定藏起来

关键模块里写清楚输入输出约定。

### 3. 不要把 magic number 到处散落

像：

- hidden dim
- patch size
- vocab size
- num_heads

最好集中管理。

### 4. 不要在很多地方重复实现同样逻辑

例如相同的 mask 构造、相同的归一化流程、相同的评估代码。

## 八、代码可维护性的一个核心标准：别人能不能快速接手

你可以用下面几个问题评价一份代码：

- 新人能否在 10 分钟内看懂主流程？
- 模型输入输出是否一眼明确？
- 训练入口是否清晰？
- 更换一个 backbone 或 loss 是否代价很大？
- 出问题时能否快速定位到模块边界？

如果答案大多是否定的，说明抽象还不够好。

## 九、从“会写”到“会设计”的升级路径

你可以把自己的成长分成几层：

### 第一层：能照着写

- 能把一个模型实现出来

### 第二层：能调通

- 能处理 shape、loss、训练问题

### 第三层：能抽象

- 能提取公共模块
- 能形成清晰接口

### 第四层：能设计系统

- 能组织数据、模型、训练、评估的边界
- 能支撑多个实验和后续扩展

CS231n 这三份作业，其实已经把你从第一层带到了第三层的门口。

## 十、你可以长期复用的一张“代码设计检查表”

以后写模型项目时，可以先问：

| 问题 | 检查点 |
| --- | --- |
| 模块边界清楚吗 | 数据、模型、loss、训练是否分开？ |
| 接口稳定吗 | 输入输出约定是否清楚？ |
| 参数管理统一吗 | 权重、状态、超参数是否分开？ |
| 复用性高吗 | 相同逻辑是否重复出现？ |
| 可调试吗 | 关键地方是否容易打印 shape 和数值？ |
| 易扩展吗 | 换模型头、换 loss、换数据是否方便？ |

## 最后一句总结

代码抽象能力与系统设计能力，会决定你以后能不能从“做作业的人”成长成“能搭建真实机器学习系统的人”。

模型当然重要，但真正让复杂项目变得可持续的，是你能否把这些模型组织成一套清晰、稳定、可扩展的代码结构。
